Section 1The design decision in one sentence

Modules become black boxes with a fixed internal approval template; the connections between modules become a versioned, editable graph stored as data.

This matches exactly what you asked for: the supervisor ↔ executive dance inside a module (submit → supervisor review → admin confirm) stays as it is — it is already consistent across the codebase and it works. What becomes dynamic is the orchestration: which module unlocks after which, in what combination (sequential, parallel, any-of), under what conditions. Today that orchestration is hardcoded Python; tomorrow it is rows in a table that an admin can edit through a visual designer. Fields can be added to a module's forms the same way — as registered schema, not as migrations.

What is already right (and must be preserved)

The codebase has strong platform genes, and the plan builds on them rather than replacing them:

• Every status change passes one gate (assert_transition) and one lock helper (fetch_site_for_update_or_404 in _common.py, using SELECT … FOR UPDATE).
• Every mutation writes audit_logs + stage_events and enqueues notification_outbox effects.
• The mirror-column pattern is a good design — it is the signalling bus between modules. It stays; only the rules that read it move from code to data.

Fig. 3 — The two-level approval template. Every module implements this today; it stays in code, per module, exactly as-is. Only the entry gate and the exit signal become platform concerns.

The consequence for the architecture: a module needs to expose only three things to the platform — (1) an entry gate it asks the platform to evaluate, (2) an exit signal it writes to the bus (its mirror column), and (3) a declared set of forms/fields. Its internals — routers, services, screens, the reject/re-upload loops — remain private code. This is why the transformation is safe: we are wrapping modules, not rewriting them.

Section 4The module graph — what becomes designable

Everything that workflow_unlocks.py and the per-service asserts do today becomes three small tables and one evaluator:

New table (schema flow)PurposeReplaces
flow_definitionsA named, versioned flow per tenant (or shared): status draft / published / retiredThe implicit single flow
flow_nodesOne row per module in a flow: module key, entry-gate condition (JSON), join semantics (all-of / any-of)_assert_design_unlocked, _assert_launched, project/NSO gates
flow_edgesDirected dependency: upstream module + the mirror value that satisfies it (e.g. legal.legal_dd_status = 'positive')design_unlock_ready() and friends in workflow_unlocks.py

Gate conditions are stored as JsonLogic expressions — e.g. {"and":[{"==":[{"var":"legal_dd_status"},"positive"]},{"==":[{"var":"finance_status"},"approved"]}]} — evaluated against the site's mirror columns by a ~50-line evaluator. Complex or data-driven gates (later, e.g. "footfall above threshold") register a named Python predicate the expression can reference, so the escape hatch is explicit, not a workaround.

Why versioning is non-negotiable

A site in Legal review must not break when the admin publishes flow v3. Each site is stamped with flow_definition_id at creation and finishes on that version. Publishing creates a new version; in-flight sites are untouched. This is the same expand-migrate-contract discipline your 44 migrations already follow, applied to workflow definitions.

Section 5Dynamic fields — extending modules "like a web"

Your second ask: add fields (or small sub-flows) to modules without vertical-slice development. The mechanism mirrors the flow mechanism — schema as data:

  1. A field registry (flow.field_definitions): module key, field name, JSON Schema fragment (type, validation, enum options), display hints, and which internal stage it appears at (executive capture vs supervisor review).
  2. One JSONB extension column per module head-table (e.g. site_details.extra, design_reviews.extra). Registered fields live there — no migration per field. Fields that later prove core get promoted to real columns in a normal migration.
  3. Validation stays server-side: Pydantic validates the extra payload against the registered JSON Schema at the router boundary — same rigor as typed columns.
  4. Forms render themselves: the frontend renders registered fields with react-jsonschema-form inside the existing module screens. Your form components in frontend/src/modules/shared become the RJSF widget theme, so dynamic fields look native.

The payoff for the flow platform: once ingested metrics exist in staging.*, a gate condition in the flow designer can reference them — "NSO opens only if projected sales ≥ X", "auto-flag approval if footfall < threshold". The two halves of this document compose.

TechnologyUsed forWhy it fits this codebase
JsonLogic
new panzi-json-logic (py)		Gate conditions as dataTiny, deterministic, no eval() risk; expressions are JSON so the flow designer can build them visually and the backend evaluates them in one function. Replaces workflow_unlocks.py logic 1:1.

new, optional		Ingestion connectorsLightweight Python EL library: handles pagination, schema inference, incremental watermarks, and loads straight into Postgres. Saves ~2 weeks of connector plumbing vs hand-rolling; runs inside your worker, no new infra.
Postgres jobs table + worker
pattern exists		Scheduling & async workYour outbox drain already proves the pattern (status/attempts/idempotent retry). A second Railway process polling catalog.jobs extends it. Procrastinate (Postgres-backed task queue) is the upgrade if job volume grows.
Supabase Storage
already in stack		Raw file landing zonestorage_service.py (service-role upload, signed URLs, bytes-outside-transactions) is reused as-is for CSV/document ingestion.
Mermaid
used in docs		Living documentationYour docs already use it; flow definitions can auto-export to Mermaid so documentation of any published flow is generated, never stale.

Deliberately rejected

Temporal / Airflow / Camunda-BPMN engines: these replace your module internals with their execution model — the opposite of the black-box decision. Your graph is small (8–15 nodes), gates are evaluated on-demand against mirror columns, and module internals stay in FastAPI. A DB-driven evaluator is ~200 lines and fully debuggable. Kafka / event buses: the mirror-column bus + outbox already gives you reliable signalling at your scale. Microservices split: the monolith with clean service boundaries is an asset here, not debt.

Phase 0 — Codify the module contract (weeks 1–2)

refactor Extract the shared write choreography (lock → assert → write mirror → audit → stage_event → outbox) from bd_service.py and siblings into one execute_transition() in _common.py. Add GET /meta/workflows serializing ALLOWED_TRANSITIONS + unlock rules; frontend consumes it and frontend/src/lib/stateMachine.js is deleted. Zero behavior change; the duplication of Fig. 2 drops from four sources to two.
Exit criteria: all module services route transitions through the helper; frontend renders transitions from the endpoint; regression suite green.

Phase 1 — Module graph as data (weeks 3–4)

new Create flow_definitions / flow_nodes / flow_edges; seed v1 as the exact current flow (including the Legal ∥ Finance all-of join). Build the JsonLogic gate evaluator; workflow_unlocks.py and the per-service asserts (_assert_design_unlocked, _assert_launched, project gate) delegate to it. Stamp sites.flow_definition_id.
Exit criteria: deleting a hardcoded gate assert changes nothing, because the evaluator answers identically from rows.

Phase 2 — Flow designer UI + versioning (weeks 5–6)

new React Flow canvas in the business-admin portal: view the live graph, clone to draft, reorder edges, edit gates via a condition builder, publish as a new version. New sites bind to the newest published version; in-flight sites finish on theirs. This is the milestone where "approval order changes without a deploy" is delivered.
Exit criteria: an admin reorders two modules in staging and a new site follows the new order while an old site is unaffected.

Phase 3 — Dynamic field registry (weeks 7–8)

new field_definitions table + extra JSONB columns on module head-tables (site_details, design_reviews, project_reviews, nso_reviews). Pydantic validation against registered schemas; RJSF rendering themed with modules/shared primitives; fields placeable at executive-capture or supervisor-review stages.
Exit criteria: a new field appears in a module form, validates, persists, and shows in audit diffs — via one inserted row.

Phase 4 — Ingestion plane (weeks 9–11)

new catalog schema (sources, syncs, jobs, datasets, health); the worker as a second Railway process; four connectors (CSV via storage_service.py, REST pull via dlt, webhook push endpoint, external Postgres). Sources UI in the admin portal.
Exit criteria: a non-developer connects a CSV and an API source from the UI; data lands in raw.*, watermarked and health-checked, with no deploy.

Phase 5 — Transforms + data-driven gates (weeks 12–13)

new Transform registry (SQL / Python) run by the worker; first pipelines (market CSV → staging; POS events → daily sales). Extend the gate evaluator so flow conditions can reference staging metrics.
Exit criteria: a published flow contains a gate that reads ingested data, end-to-end.

Phase 6 — Hardening (week 14)

polish Catalog/lineage UI, auto-export of published flows to Mermaid docs, load-test the evaluator, retire dead code paths (contract step of expand-migrate-contract).

Fig. 8 — The strangler sequence: at no point does an old route break; each mechanism is retired only after its replacement has answered identically in production.

Three disciplines carry the whole plan, and all three already exist in the repo: (1) expand-migrate-contract migrations (backend/database/migrations/, 44 so far); (2) the regression test suite (backend/tests/, ~35 files) as the safety net for every service that moves onto execute_transition(); (3) version stamping, which isolates every in-flight site from every published change — the same principle protects flow definitions, field definitions, and transform definitions alike.

Section 10Open questions to discuss

QuestionWhy it matters now
Are flows per-tenant, or shared templates with tenant overrides?Decides whether flow_definitions.tenant_id is nullable and how the designer's "clone" works. Cheap now, expensive to retrofit.
Join semantics: is any-of needed at launch, or only all-of?Today's only join (Legal ∥ Finance → Design) is all-of. Supporting any-of from day one costs little in the schema; adding it later touches the evaluator and the designer.
Can an admin edit a draft flow's gates freely, but only append modules to published flows?Guardrails on the designer determine how much validation the publish step needs (cycle detection, unreachable-module checks, orphaned-mirror checks).
Which module gets the first dynamic field in production?Recommend site_details (BD capture) — highest field-request churn, lowest risk, and completion_pct logic will need a rule for extra fields.
Worker hosting: second Railway service vs. in-process scheduler first?Second process is cleaner (Rubix "design for disruption" principle) but adds ~$; in-process APScheduler is an acceptable week-9 shortcut if budget-constrained.
